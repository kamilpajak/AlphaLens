# P2 — TEST STRATEGY (TDD + integration) for the per-exchange session VIEW

**Status:** DRAFT — this is a PLAN. P2 is NOT built yet. Every test below is the **red-first spec**: it must be writable and committable BEFORE the production code, assert behaviour/contract (never an impl detail), and fail until the code that satisfies it ships.
**Date:** 2026-06-05
**Specifies tests for:** [`news_lake_p2_session_view_design_2026_06_05.md`](news_lake_p2_session_view_design_2026_06_05.md) (the P2 design — DRAFT, gated by the adversarial review's 4 unresolved findings)
**Depends on:** the P1c **lake-raw substrate fix** (shipped on this branch) — the lake now stores the RAW per-source union (`news_ingest._write_lake_run`); without it no P2 test can run. The P1 bitemporal lake (`multi_exchange_news_lake_design_2026_06_05.md`) is a HARD GATE for P2b.

**Why a money-entrusted tool needs this rigour:** people act on these briefs with real capital. The session-VIEW change re-homes ~10-15% of news between adjacent briefs (design §6). A silent off-by-one at the window seam (late arrival dropped, after-hours print landing in the wrong brief, a populated-Postgres PK migration that loses rows) is a wrong answer served to someone deploying money. The 4 gating tests exist to make each of those failure modes RED-first so the build can only go green by fixing it.

## Adversarial review outcome + revisions (2026-06-05)

A 4-lens review (findings-coverage, tdd-writability, e2e-realism, completeness) confirmed **TDD-writability SOUND** (interfaces concrete enough to write red-first, behavioural not structural) but flagged the plan **flawed** on 3 lenses. The body below stands; these revisions AMEND the named tests. The biggest is a **P2 design refinement** the test-planning surfaced — it belongs back in the design memo before P2 is built.

**R1 (HIGH — design refinement). `lake_view.py` must REPLAY the full reduction pipeline, not just window + id-collapse.** The raw lake (post #461) stores the per-source union **before** url-canon dedup, **before** lexical clustering, and with **no `_cluster_rank_ts`** column (computed inside `_cluster_same_day_lexical`/`_allocate_per_source`, dropped before write). Consequences for the tests:
- The cap is **`_allocate_per_source(merged, _SOURCE_QUOTA_WEIGHTS, max_items)`** — a per-source QUOTA (`ceil(max*weight)` + recency backfill), **not** flat `head(200)`. The `test_reapplies_200_cap_after_collapse` / `test_cap_keeps_newest_by_cluster_rank_not_oldest` tests must model the real quota (assert SEC/Polygon are NOT starved under GDELT volume — the exact crowding the quota prevents), and must NOT sort by a `_cluster_rank_ts` that does not exist on lake rows (the read-view recomputes it or falls back to `timestamp`).
- The by-`id` collapse alone does **not** reproduce `{D}.parquet` — url-canon + lexical clustering are lost in the raw substrate. The read-view must re-run **url-canon dedup → lexical cluster → quota-cap** over the windowed (possibly 2-partition) union. Add `test_read_view_reruns_urlcanon_and_lexical_clustering` (a same-story-distinct-url pair the raw lake keeps collapses in the view); the P2a inertness pin `test_lake_view_reproduces_utc_day_parquet` only passes if the view drives the full replay. **`lake_view` = window-filter + as-of-collapse + replay(url-canon, lexical-cluster, per-source-quota-cap).**

**R2 (HIGH). Per-source-quota × cap over the 2-partition union is a NEW P2b code path, not deferrable to P1.** The cap is re-applied on the session window (unions ~2 `session_date` partitions, a wider valid-time range, more rows than either UTC day). Add an explicit P2b `test_quota_cap_over_two_partition_union` — do not defer to "P1's test".

**R3 (MEDIUM). No-historical-lake test is too weak — drop the empty-frame disjunct.** `test_as_of_before_lake_inception_returns_empty_or_unavailable` currently passes the very silent-empty impl the finding forbids (an empty partition glob returns empty, satisfying the OR). Fix: pre-inception **must raise `LakeUnavailable` / return `available=False`** (empty is indistinguishable from a real empty in-era session). And `test_lake_inception_lower_bound_is_pinned` must assert `read_session_view` **consults** the constant, not merely that it exists. Add an exactly-at-`LAKE_INCEPTION_DATE` boundary control (`>=` vs `>` off-by-one).

**R4 (MEDIUM). Mark the P2a→P2b transition as the faithful-replay lower bound.** Add `test_pre_p2b_replay_diverges_is_known_bound`: a `session_date` AFTER lake inception but whose brief was generated by the old `{D}.parquet` path (the P2a no-op era) — lake-replay (raw union) ≠ served brief (capped view); pin this as a KNOWN bound, not a bug, so a future replay over the P2a window does not silently claim fidelity.

**R5 (MEDIUM). Extend the gap-free property grid across the seams, not one ordinary day.** Run the enumerated every-instant-in-exactly-one-window grid over the **DST-transition 48h** (spring-forward AND **fall-back Nov 1 2026**, the ambiguous 01:00–02:00 local — currently unmentioned) and the **Black-Friday early-close day**, not only 2026-06-03. Add a flagship-uniqueness test (no two cuts claim the same `session_date`+role) across a holiday-shortened week.

**R6 (MEDIUM, e2e). Close the served-endpoint seam + add a straddle to the golden.** D.1's golden chain stops at the brief parquet; C.2's endpoint test reads a hand-populated db, NOT the pipeline output. Add to D.1 one **parquet → `rebuild_briefs_cache` → `GET /v1/brief/<date>`** assertion on session-window content (close the parquet→Postgres→served seam), and one **midnight-straddle** cut driven end-to-end (the cross-partition read is currently unit-only).

**R7 (LOW). Split a "pin the decision first" sub-slice + add negative controls.** Tests that import an undecided API surface (`allowed_lateness` param vs watermark column for C.1; `date→(MIC,date)` map vs deprecate for C.2) are red for the WRONG reason (ImportError/undecided signature), breaking the red-first contract — sequence a tiny decision-pinning slice first. Add the missing negative controls: a date-anchored Polygon row inside `[00:00Z, 20:30Z)` that appears in BOTH the UTC-day and session-window labellings (the content-diff D.2 fraction-band 0.05–0.25 is loose enough to pass even if the WRONG rows move).

## C. THE 4 GATING TESTS (the heart — each RED-first, only green once P2 fixes it)

Each maps 1:1 to an "unresolved before any P2 build" finding (design §10 header). The build is **gated**: these must be committed RED before the corresponding production code.

### C.1 LATE-ARRIVAL re-admission — `tests/thematic/test_lake_view_late_arrival.py`

**Finding 1.** A row valid-in-window but `ingested_at > this_cut` is dropped at the valid/txn seam unless the allowed-lateness rule re-admits it. The inverse of dedup: dedup operates WITHIN a window, watermarking operates ACROSS windows.

**Fixture:** two `_run_ingest` calls.
- Run A at `now = 2026-06-03 20:30Z` (the cut) writes id `late-1` with `timestamp = 2026-06-03 19:00Z` (valid IN the flagship window `[16:30Z, 20:30Z)`) — but suppose source latency means this row only appears in **Run B**.
- Run B at `now = 2026-06-03 20:45Z` (after the cut) writes `late-1` (`timestamp = 19:00Z`, `ingested_at = 20:45Z`).

**Test methods:**
- `test_late_arrival_excluded_at_strict_cut_as_of` — `read_session_view("XNYS", 2026-06-03, as_of=20:30Z)` does NOT contain `late-1` (strict `ingested_at <= as_of` at the bare cut). This pins the BASELINE seam behaviour.
- **`test_late_arrival_readmitted_within_allowed_lateness`** — `read_session_view("XNYS", 2026-06-03, as_of=20:30Z, allowed_lateness=timedelta(minutes=30))` (or the chosen re-admission API) DOES contain `late-1`, because `19:00Z` is in-window and `20:45Z <= cut + lateness`. **This is the gating assertion — RED until the watermarking rule ships.**
- `test_late_arrival_does_not_shadow_in_window_id` — if id `late-1` ALSO had an earlier in-window copy (`ingested_at = 20:10Z <= cut`), the as-of collapse keeps `max(ingested_at)`; assert the late copy does NOT silently shadow the in-window one when lateness is OFF (the late row is SKIPPED, not promoted over the timely one). Pins the finding-1 note "late arrival is SKIPPED (not shadowed)".

**Blocking infra gap:** Late-arrival watermarking / allowed-lateness is a P1 corrective (req #5) with NO implementation yet — only a design. The test mocks lake rows with `ingested_at > as_of` and asserts the re-admission boundary; **it cannot pass until the lateness gate is built** (that is the point — it is the red spec). If the lateness API surface is undecided, the test pins the chosen surface (param name `allowed_lateness` vs a watermark column) and must be updated when that decision lands.

### C.2 POSTGRES PK MIGRATION on a POPULATED db — `briefs/tests/test_brief_pk_migration.py`

**Finding 2.** `Brief` PK `(date, ticker)` → `(exchange, session_date, ticker)`, `DayMeta` `date` → `(exchange, session_date)`: a drop-and-recreate on populated Postgres (the #331/#340/#372 skew class). Touches ~15 date-keyed read sites incl. `/v1/brief/<date>`, feedback POST, `edge/ingest`.

Three named tests (Postgres-only, `skipIf connection.vendor != "postgresql"`, runs in the CI `django` job's `postgres:16-alpine` service):

- **`test_migration_skew_guard_catches_pk_change` (migration-skew-guard test)** — model the #372 pattern: roll DB back to the pre-P2 migration (PK = `(date, ticker)`) while code is at HEAD (new PK), then call `migration_guard.assert_schema_current()` / `rebuild_briefs_cache` and assert it RAISES the skew error (both directions: DB-behind via `migration_plan()`, DB-ahead via `recorded - graph`). Reuses `briefs/migration_guard.py` + the `test_migrate_skew.py` rollback harness. Extends it from the column-rename case to the **PK-change** case.
- **`test_repk_preserves_existing_rows` (data-preservation test)** — pre-populate Postgres with N real `Brief` rows + M `DayMeta` rows at migration `0006` (old PK), run the P2 migration forward, assert: (1) all N Brief rows still present and queryable by `(exchange, session_date, ticker)` with `exchange="XNYS"`, `session_date == date`, `ticker` unchanged; (2) all M DayMeta rows present, keyed `(exchange, session_date)`, exactly one row per tuple (no duplicate, no drop). This is the "drop-and-recreate must not lose rows" guarantee — the money-entrusted core.
- **`test_brief_endpoint_contract_after_repk` (endpoint-contract test)** — after migration, `GET /v1/brief/<date>` (and `/v1/days/{date}`, `/v1/days/{date}/candidates`) still resolves: one legacy `date` now maps to N `(exchange, session_date)` rows. Assert the transition strategy (design §3.1 retains `date` as a plain column) — `date=D` resolves to `(exchange="XNYS", session_date=D)`. Pins whichever backward-compat path is chosen (map `date → (XNYS, date)` vs explicit deprecation). Envelope shape `{data, meta}` unchanged.

**Fixture each needs:** the CI `postgres:16-alpine` service (`DATABASE_URL=postgres://...`, exists). A **new** end-to-end populated-Postgres fixture (`_populate_briefs_at_migration_0006`) — the discovery flags this gap: `test_migrate_skew.py` currently uses an EMPTY parquet and only reproduces the `bulk_create` insert-path failure; the data-preservation + endpoint tests need REAL Brief/DayMeta rows persisted at the old migration, then migrated forward, then SELECT/filter/aggregate asserted on the new schema. The migration file itself does not exist yet (gap) — these tests pin both the fresh-schema and upgrade paths.

### C.3 NO-HISTORICAL-LAKE bound — `tests/thematic/test_lake_view_inception.py`

**Finding 3.** The lake began 2026-06-05. As-of replay for any pre-P2 `session_date` has no data and must return empty / declared-unavailable — never a silent wrong answer.

**Test methods:**
- **`test_as_of_before_lake_inception_returns_empty_or_unavailable`** — `read_session_view("XNYS", session_date=2026-05-20, as_of=...)` (before inception) returns an empty `NEWS_COLUMNS` frame OR raises a declared `LakeUnavailable`-style error — assert it does NOT return a partial/wrong frame silently. Pin the lower bound: define `LAKE_INCEPTION_DATE` and assert queries below it are explicitly unavailable.
- `test_as_of_after_inception_returns_rows_up_to_as_of` — a `session_date` after inception returns the in-window rows known at `as_of` (the normal path; control case proving the bound is not over-broad).
- `test_lake_inception_lower_bound_is_pinned` — assert the inception constant exists and equals the documented lake start, so a future replay cannot silently scan an empty directory and report "0 news" as if that were the real session.

**SPA/API zero-row note (web-side, flagged not built here):** the design (finding 3) also requires the SPA not to crash on a zero-row brief response. That is a web (`apps/web`) test, NOT Python — recorded in §E as a gap. The Python tests here pin only the server-side empty/unavailable contract.

**Blocking infra gap:** depends on `lake_view.py` + an inception constant that does not exist yet.

### C.4 PER-WINDOW DEDUP — `tests/thematic/test_lake_view_dedup.py`

**Finding 4.** The same story ingested across two runs/partitions can survive with distinct `id`s in the RAW lake; the read VIEW must re-dedup at query time by `id`, keeping `max(ingested_at)`. Raw lake keeps both; view dedups.

**Fixture:** synthesize TWO lake run files (the gap the discovery flags — no existing fixture shows the same story twice with distinct ingested_at).
- Run 1 at `now=T1` writes id `story-A` (`ingested_at=T1`).
- Run 2 at `now=T2 > T1` writes id `story-A` again, same content, `ingested_at=T2` (re-cluster on replay can also yield a distinct id `story-A'`; cover both: same-id and same-content-distinct-id).

**Test methods:**
- **`test_same_id_two_runs_collapses_to_one_row_max_ingested_at`** — `read_session_view` returns exactly ONE `story-A` row, the one with `ingested_at == T2`.
- `test_raw_lake_keeps_both_runs` — assert the underlying lake partition still holds BOTH run files (the VIEW dedups; the substrate does not lose the audit copy) — read the run files directly via `_run_files`.
- `test_id_appearing_n_times_keeps_only_freshest` — id present in N≥3 runs → output has 1 row, `ingested_at == max`.
- `test_distinct_ids_same_content_are_not_collapsed_by_view` — pin the boundary: the id-keyed collapse does NOT merge `story-A` and `story-A'` (distinct ids) — that is cross-listing/global-event dedup (§4, P3), NOT the as-of collapse. Assert both survive (documents the deliberate scope line).

**Fixture each needs:** a **new** `_write_two_runs_same_story(lake_dir, *, t1, t2)` helper (synthesize two `run=*.parquet` under one `session_date=` partition) — flagged as a gap (no existing fixture).
